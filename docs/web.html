2. Principe général : structure de données et mots principaux (addattr, addcontent)

Structure de données

La seule structure de données manipulée par le package 'web' est la suivante :

{ (c'est un dictionnaire)
"tag" : "my_tag", (corresponds au nom du tag : html, head, body, ...)
"content" : list_of_content, (corresponds au contenu du container représenté par le tag)
"attrs" : attr_dict, (dictionnaire des attributs du tag)
"container" : "y|n" (y = c'est un container, n = ce n'est pas un container)
}

Exemple de structure de données définie avec Beetle :

{ tag : "input" , content : [ ] , attrs : attrs @ , container : "n" }

Cette structure pour définir un input HTML. Ce n'est pas un container donc il n'a pas de contenu et il n'a que des attributs. Les attributs se retrouvent dans la variable attrs.

Le package se charge de créer les tags principaux de la page web : comme 'html', 'head', 'body', 'title'. Ces structures sont stockées dans des variables qui portent le même nom : la structure pour le tag 'html' se trouve dans la variable html, etc...

Mots principaux

Pour ajouter un attribut à un tag, ce sera le mot : addattr. La syntaxe du mot est la suivante :

contenant_variable "attribut_name" "attribut_value" addattr

Pour ajouter un contenu à un tag, ce sera le mot : addcontent. La syntaxe du mot est la suivante :

contenant_variable { structure | Chaine_de_caractères } addcontent

Exemple concret (vous pouvez voir que ces 2 mots sont très importants dans le package 'web') :

<?btl
webreset
body form addcontent
title "Page test de passage de paramètres dans un formulaire" addcontent
form "action" "test_form_result.btl" addattr
html "lang" "en" addattr
head "UTF-8" charset addcontent
head "userarea/css/file.css" headlink addcontent
form "Ceci est mon label : " "mydiv" label addcontent
"mydiv" "fred" "text" fieldarea var fa drop
fa "name" "faname" addattr
form fa addcontent
form br addcontent
form "submit" "Valider" button addcontent
html @ generate cls
forget fa
?>

3. La construction du HTML : widgets de base (a, abbr, address, area, ...)

Ce package vous propose de vous créer cette structure pour chaque tag HTML. Avec les 2 mots décrits ci-dessus, vous pouvez en créer de plus complexes. Ceux qui vous sont proposés dans la suite représentent les tags basiques.
HTML est un langage permisif donc si vous positionnez un tag au mauvais endroit, rien ne se passera. HTML n'en tiendra pas compte. Du coup, Beetle ne vous emchêche pas de faire n'importe quoi. La charge du contrôle est laissé à HTML.
Voici donc la description de cette liste exhaustive de mots :

Mot 'a'

Tag : <a>

Attributs obligatoires : href

Contenu : chaine de caractères

Container? : oui

La syntaxe de ce mot est :

"content" "href_attribut" a

--> {'tag':'a', 'content':['content'], 'attrs':{'href':'href_attribut'}, 'container':'y'} est positionné sur la pile
--> <a href="href_attribut">content</a>

Mot 'layoutgrid'

Description : ce mot vous permet de renseigner le css général de votre layout. Ce css se retrouve dans la variable laystyle.

Paramètres : "nom_de_la_ressource_css" "valeur_de_la_ressource" : ce sont des chaines de caractères

  • Le nom_de_la_ressource_css peut prendre les valeurs suivantes : template-columns, template-rows, column-gap, row-gap ou other
    Dans tous les cas sauf pour other, Beetle ajoute 'grid-' avant le paramètre dans le css. Cela permet à Beetle de contrôler un minimum de cohérence dans le nom de l'attribut css
    Pour other, cela vous permet de renseigner du css général au layout. Dans ce cas-la, il faut préciser la ressource et sa valeur que vous souhaitez affecter
  • valeur_de_la_ressource représente la valeur de nom_de_la_ressource_css. C'est ce que vous avez besoin que ce soit

Syntaxes :

"nom_de_la_ressource_css" "valeur_de_la_ressource" layoutgrid

"other" "nom_de_la_ressource_css: valeur_de_la_ressource" layoutgrid

Exemple :

"template-columns" "200px repeat(4, 1fr)" layoutgrid ( va créer la ressource css : grid-template-columns avec la valeur 200px repeat(4, 1fr) )
"template-rows" "50px 500px 50px" layoutgrid ( va créer la ressource css : grid-template-rows avec la valeur 50px 500px 50px )
"column-gap" "1px" layoutgrid ( va créer la ressource css : grid-column-gap avec la valeur 1px )
"row-gap" "1px" layoutgrid
"other" "background-color: indianred" layoutgrid ( va créer la ressource css : background-color: indianred )
"other" "padding:10px" layoutgrid ( va créer la ressource css : padding:10px )

Mot 'layoutarea'

Description : vous permet de définir le css de chaque zone de votre layout. Ce css se retrouve dans la variable laystyle.

Paramètres : "nom_de_la_ressource_css" numéro_de_la_zone "valeur_de_la_ressource"

Syntaxe :

"area" numéro_de_la_zone "valeur_de_la_ressource" layoutarea

"other" numéro_de_la_zone "nom_de_la_ressource_css: valeur_de_la_ressource" layoutarea

Exemple :

"area" 0 "1 / 1 / 2 / 6" layoutarea ( va créer la ressource grid-area avec la valeur 1 / 1 / 2 / 6 pour la zone 0 )
"area" 1 "2 / 1 / 3 / 2" layoutarea
"area" 2 "2 / 2 / 3 / 6" layoutarea
"area" 3 "3 / 1 / 4 / 6" layoutarea
"other" 0 "background-color:white" layoutarea ( va créer la ressource background-color:white pour la zone 0 )
"other" 0 "display: grid" layoutarea
"other" 0 "place-items: start center" layoutarea
"other" 1 "background-color:white" layoutarea
"other" 2 "background-color:white" layoutarea
"other" 3 "background-color:white" layoutarea
"other" 3 "display: grid" layoutarea
"other" 3 "place-items: end center" layoutarea

Description : permet de regrouper les définitions css qui ont les mêmes sélecteurs. ce mot peut être utilisé à tout moment.

Cela indique à votre navigateur qu'il va devoir afficher une page HTML.

8. La gestion des sessions et les redirections : (session, getsession, setsessionduration, ...)

Plusieurs choses sont à prendre en considération pour comprendre le fonctionnement des sessions Beetle. Avec Beetle, vous pouvez :

  1. Créer une session et faire une redirection
  2. Créer une session sans faire de redirection
  3. Créer une page sans session et faire une redirection
  4. Créer une page sans session ni redirection

Les composantes d'une session sont les suivantes :

  1. Un cookie coté client est mis en place au moment de la création de la session avec une durée de validité
  2. Un fichier coté serveur portant le nom de la session et contenant les variables de sessions que vous aurez définies

Le mot 'session' est le seul mot vous permettant de créer une session. Il va créer un cookie au niveau du client et un fichier de session sur le serveur contenant les variables de sessions.
Par exemple, pour indiquer qu'une authentification est réussie dans votre application, vous devrez utiliser les variables de session.
Si vous n'avez pas de variable de session à positionner dans votre application, le fichier de session contiendra '{}'
Le cookie coté client possède une date de validité qui est fixée à 30 minutes à partir de sa création et contient pour sa valeur un identifiant de session qui est utilisé pour le nom du fichier de session. Cela permet de pouvoir supprimer l'ancien fichier de session s'il existe pour en créer un nouveau. Ce mécanisme est en place pour éviter de stocker tous les fichiers de session même ceux des sessions périmées.

La syntaxe du mot 'session' est la suivante :

"noredirect | redirect" "nom_de_session" session

Pour pouvoir faire une redirection après la création d'une session, vous devrez utiliser le paramètre 'redirect'. L'entête de la réponse du serveur n'est pas la même que pour une page classique.

Du coup, pour faire une redirection, ce sera le mot 'redirect' qui vous permettra de le faire.
Attention : il ne faut rien faire après la redirection au risque d'empêcher le serveur de répondre correctement à votre requête HTTP.

La syntaxe du mot 'redirect' est la plus simple possible :

"url_de_redirection" redirect

Maintenant, il vous faut pouvoir récupérer la session valide en cours. Pour cela, nous avons un autre mot 'getsession' dont la syntaxe est la suivante :

"nom_de_session" getsession

Voici un exemple pour illustrer tout cela :
Le point de départ de notre exemple est le contenu de notre fichier 'test_form.btl' qui contient un formulaire.

<?btl
webreset
body form1 addcontent
title "Page test de passage de paramètres dans un formulaire" addcontent
form1 "action" "test_form_result.btl" addattr
html "lang" "en" addattr
head "UTF-8" charset addcontent
head "userarea/css/file.css" headlink addcontent
form1 "Ceci est mon label : " "mydiv" label addcontent
"mydiv" "fred" "text" fieldarea var fa drop
form1 fa addcontent
form1 br addcontent
form1 "submit" "Valider" button addcontent
html @ generate cls
forget fa
?>

L'action de notre formulaire pointe sur le fichier 'test_form_result.btl' qui contient :

<?btl
"redirect" "beetle" session ( création d'une session qui se nomme 'beetle' )
"http://localhost/pybee/test_session.btl" redirect ( redirection vers 'test_session.btl' )
?>

Et 'test_session.btl' récupère la session et affiche le tableau des variables de session :

<?btl
htmlcontent ( c'est une simple page et donc on affiche la bonne entête )
"beetle" getsession . ( récupère la session 'beetle' et affiche les variables de session )
?>

Concernant les sessions, quelques mots supplémentaires vous sont mis à disposition :

  • session? : vous indique qu'une session à été créée (mot sans paramètre)
  • setsessvar : vous permet de positionner une variable de session dont la syntaxe est :
    "key" "value" setsessvar
  • getsessvar : vous permet de lire une variable de session dont la syntaxe est :
    "key" getsessvar
  • setsessduration : positionne une durée de validité pour votre session - 30 minutes par défaut dont la syntaxe est :
    duration setsessduration
  • sessduration? : vous indique la durée de validité d'une session (mot sans paramètre)
